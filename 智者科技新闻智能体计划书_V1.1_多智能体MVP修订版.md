## 1. 项目概述

智者是一个面向每日科技新闻收集、筛选、排序和摘要的多智能体系统。它的核心目标不是简单堆砌新闻链接，而是帮助用户在短时间内理解当天科技领域真正重要的变化。

智者每天自动从多个科技信息源采集内容，包括 Hacker News、GitHub、arXiv、RSS、科技媒体、公司官方博客等。系统会通过多个智能体协作完成内容采集、清洗、去重、聚类、分类、排序、摘要、来源校验和日报生成，最终输出一份结构清晰、重点明确、带有来源链接的每日科技简报。

V1.1 的调整重点是：保留多智能体架构并同步开发，但收缩 MVP 范围。第一版不追求完整大而全产品，而是优先跑通“多源采集 -> 多智能体处理 -> Markdown 日报输出 -> 定时运行”的闭环。

## 2. 产品目标

1. 每天自动生成一份 5 分钟可读完的科技新闻简报。
2. 对新闻进行重点排序，让用户先看到最重要的内容。
3. 对每条重点新闻说明“发生了什么”“为什么重要”“可能影响谁”。
4. 保留来源链接，减少幻觉和误读。
5. 保留多智能体分工模式，使系统后续容易扩展。
6. MVP 阶段优先支持 Markdown 日报输出。
7. Beta 阶段支持人工审核、置顶、隐藏、改写和权重调整。
8. 后续支持个性化关注领域，例如 AI、芯片、开发者工具、网络安全、开源项目、公司动态等。

## 3. 设计原则

### 3.1 保留多智能体，但控制复杂度

智者保留 Collector、Cleaner、Clusterer、Classifier、Ranker、Summarizer、Verifier、Editor、Publisher 等角色。但在 MVP 阶段，这些角色不一定都需要复杂的大模型自主规划能力，而是先以“轻量 Agent 模块”的形式实现。

每个 Agent 都应具备统一接口：

```text
输入：上一个 Agent 的标准化输出
处理：规则、工具调用或 LLM 调用
输出：结构化结果 + 日志 + 错误信息
```

这种方式既保留多智能体架构，又避免一开始陷入复杂编排和调试困难。

### 3.2 先做可运行闭环，再做高级能力

MVP 的核心不是一次性完成所有新闻源、所有发布渠道和所有质量评估，而是先完成每天稳定生成日报的闭环。

MVP 闭环：

```text
采集候选新闻
-> 标准化 NewsItem
-> 清洗和去重
-> 分类和排序
-> 摘要和影响分析
-> 来源链接保留
-> Markdown 日报生成
-> GitHub Actions 定时运行
```

### 3.3 模型负责理解和生成，工具负责真实操作

大模型主要用于分类、排序解释、摘要和影响分析。新闻采集、数据库存储、链接去重、文件输出、定时运行等操作应由确定性代码完成。

### 3.4 来源引用优先于复杂事实校验

MVP 阶段不追求完整自动事实校验系统，而是先实现来源引用与可信度标记。每条 P0/P1 新闻必须保留来源链接，摘要不得编造来源中不存在的信息。

## 4. 参考范式

智者可以参考目前常见智能体系统的组织方式：

1. 多智能体分工模式：参考 CrewAI 中 agents、tasks、crews 的设计，将采集、筛选、排序、摘要、校验、发布拆成不同角色。
2. 流程编排模式：参考 LangGraph 的有状态工作流，让每天的新闻处理流程可恢复、可追踪、可插入人工审核。
3. 工具调用模式：参考 OpenAI Agents SDK 的工具、护栏、handoff、tracing 思路，让模型负责判断和生成，让工具负责抓取、存储、发布。
4. 新闻摘要机器人模式：参考常见 AI newsletter、HN digest、RSS summarizer 的输出形态，但增加更强的优先级排序和影响分析。

MVP 阶段不强制依赖某个框架。可以先用 Python 类和 Orchestrator 实现多智能体流程，后续再迁移到 LangGraph、CrewAI 或 OpenAI Agents SDK。

## 5. 用户场景

### 5.1 每日晨报

用户每天早上收到一份科技简报，快速了解过去 24 小时内最重要的科技新闻。

### 5.2 重点追踪

用户关注 AI、芯片、开源工具、安全漏洞等主题，智者会自动提高相关内容权重。

### 5.3 决策辅助

当出现重大新闻，例如模型发布、法规变化、安全漏洞、公司战略调整时，智者不仅摘要内容，还解释潜在影响。

### 5.4 内容沉淀

每日新闻可以沉淀为 Markdown、GitHub Issue、网页、Notion 或知识库，方便后续搜索和复盘。

## 7. 智能体角色设计

### 7.1 MVP 保留的智能体

| 智能体角色 | 责任 | 输入 | 输出 | MVP 实现方式 |
|---|---|---|---|---|
| 调度者 Orchestrator | 按顺序调度各 Agent，记录状态和错误 | 配置文件、运行时间 | 流程结果、日志 | Python 主流程 |
| 采集者 Collector | 抓取多源新闻 | RSS、API、网页 | 原始候选新闻 | 规则和 API 调用 |
| 清洗者 Cleaner | 清理无效内容 | 原始新闻 | 标准化新闻对象 | 规则处理 |
| 聚类者 Clusterer | 去重和事件聚类 | 标准化新闻 | 新闻事件组 | URL + 标题相似度 |
| 分类者 Classifier | 判断主题与实体 | 新闻事件组 | 分类和标签 | 规则 + LLM 可选 |
| 排序者 Ranker | 计算新闻重要性 | 分类新闻 | 优先级列表 | 规则评分 + LLM 解释 |
| 摘要者 Summarizer | 生成简明摘要 | 高优先级新闻 | 摘要和重点 | LLM |
| 校验者 Verifier | 检查来源和可信度 | 新闻、来源链接 | 来源引用和风险标记 | 规则优先 |
| 编辑者 Editor | 生成日报结构 | 排序新闻 | 日报草稿 | 模板生成 |
| 发布者 Publisher | 分发内容 | 日报草稿 | Markdown 文件 / GitHub Issue | 文件输出起步 |

### 7.2 Beta 后再强化的智能体

| 智能体角色 | 责任 | 推迟原因 |
|---|---|---|
| 反馈分析者 Analyst | 学习人工调整，优化权重 | 需要积累人工反馈数据 |
| 深度分析者 Deep Analyst | 对重大事件生成专题分析 | 成本较高，适合 V1.2 |
| 知识库管理员 Knowledge Curator | 将日报沉淀为可搜索知识库 | 需要稳定日报数据后再做 |

## 8. 多智能体运行流程

MVP 的多智能体流程如下：

```text
Scheduler / GitHub Actions
  -> Orchestrator
  -> Collector Agent
  -> Cleaner Agent
  -> Clusterer Agent
  -> Classifier Agent
  -> Ranker Agent
  -> Summarizer Agent
  -> Verifier Agent
  -> Editor Agent
  -> Publisher Agent
```

每次运行都生成一个 run_id，并保存以下内容：

```text
运行时间
每个 Agent 的输入数量
每个 Agent 的输出数量
每个 Agent 的错误信息
最终日报路径
LLM 调用次数
大致成本
```

## 11. 重点排序模型

新闻总分建议采用加权评分：

```text
总分 =
0.25 * 行业影响力 +
0.20 * 用户相关性 +
0.15 * 时效性 +
0.15 * 来源可信度 +
0.10 * 新颖性 +
0.10 * 可行动性 +
0.05 * 讨论热度
```

### 11.1 优先级定义

| 等级 | 含义 | 示例 |
|---|---|---|
| P0 | 今日必须知道 | 重大模型发布、严重安全漏洞、关键监管政策 |
| P1 | 高价值重点 | 大厂产品更新、重要开源项目、重要论文 |
| P2 | 可选阅读 | 普通融资、产品小更新、行业评论 |
| P3 | 观察归档 | 低可信、重复报道、营销稿 |

### 11.2 排序解释字段

每条重点新闻需要保留排序理由：

1. 为什么这条新闻重要。
2. 它影响哪些人或行业。
3. 它是否来自可靠来源。
4. 是否有多个来源交叉验证。
5. 是否值得用户采取行动。

### 11.3 MVP 排序策略

MVP 不要求一开始就完全依赖 LLM 排序。建议采用两段式：

```text
第一段：规则粗排
- 来源可信度
- 发布时间
- HN 分数 / GitHub stars / arXiv 主题
- 是否命中重点关键词

第二段：LLM 精排
- 只对 Top 30-50 候选调用 LLM
- 输出重要性理由、影响对象和栏目建议
```

这样可以降低成本，也能减少低质量新闻进入摘要环节。

## 12. 每日内容安排

MVP 日报控制在 10-20 条新闻，阅读时间控制在 5 分钟以内。

| 栏目 | MVP 数量 | 内容要求 |

## 18. Prompt 与模型策略

### 18.1 分类 Prompt

分类时要求模型输出结构化 JSON，包括主分类、标签、涉及公司、涉及技术、可信度提示。

### 18.2 排序 Prompt

排序时要求模型不要只看热度，还要解释影响力。输出必须包含：

1. 重要性分数。
2. 重要性理由。
3. 影响对象。
4. 是否需要进入今日最重要栏目。

### 18.3 摘要 Prompt

摘要时禁止泛泛而谈，要求：

1. 先说事实。
2. 再说影响。
3. 保留不确定性。
4. 不编造来源中没有的信息。
5. 每条重点新闻必须保留来源链接。

### 18.4 Verifier Prompt

Verifier 不直接判断复杂真假，而是输出来源状态：

1. 是否有原始来源。
2. 是否有多个来源。
3. 是否来自官方博客、论文、媒体或社区。
4. 是否存在明显不确定性。
5. 是否应降低优先级。

## 21. 风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 多智能体调试复杂 | 某个 Agent 出错导致全流程中断 | 每个 Agent 保存输入输出和错误日志；Orchestrator 支持跳过非关键 Agent |
| 信息源噪声高 | 热点不等于重要 | 增加可信度和影响力权重 |
| 重复报道过多 | 日报内容冗余 | 做事件聚类和来源合并 |
| 摘要幻觉 | 模型编造不存在的信息 | 强制引用来源，摘要只基于 clean_text 和 source_url |
| 排序不符合用户偏好 | 重要新闻被漏掉 | 增加人工反馈和权重配置 |
| API 成本过高 | 每日调用过多 | 先规则过滤，再对高候选调用 LLM |
| 来源版权风险 | 复制大段正文 | 只保存摘要、链接和必要元数据 |
| 抓取失败 | 日报缺失 | 加重试、备用源、失败告警 |
| 新闻时效不稳定 | 抓取时间不同导致内容缺失 | 固定 24 小时时间窗并记录 run_id |

## 22. MVP 验收标准

V1.1 MVP 的成功标准：

1. 系统保留多智能体架构，至少包含 Collector、Cleaner、Clusterer、Classifier、Ranker、Summarizer、Verifier、Editor、Publisher 九个 Agent。
2. 每个 Agent 具备统一 run(input) 接口，并能被 Orchestrator 顺序调用。
3. 每天稳定生成一份科技新闻日报。
4. 日报包含 10-20 条高质量新闻。
5. Top 5 新闻中至少 3-4 条被人工认为重要。
6. 每条 P0/P1 新闻都有来源链接。
7. 每条重点新闻都有“为什么重要”的解释。
8. 日报阅读时间控制在 5 分钟内。
9. 系统可以通过配置调整新闻源、栏目和排序权重。
10. GitHub Actions 可以定时运行并保存日报文件。
